<objective>
Create the snapshot and notification cleanup automation. A Postgres function handles the per-card retention logic via SQL (since Supabase JS cannot do DELETE with JOIN), and a lightweight Node.js script calls it via RPC. A daily GitHub Actions workflow triggers the cleanup. Also add a concurrency guard to the existing fetch-prices workflow.

Purpose: Old price snapshots and notifications accumulate indefinitely without cleanup, wasting storage and slowing queries. Per-card retention_days (default 30) keeps data lean.
Output: Migration file, cleanup script, cleanup workflow, updated fetch-prices workflow.
</objective>

  <files>supabase/migrations/00004_automation.sql, scripts/cleanup-snapshots.ts</files>
  <action>
1. Create `supabase/migrations/00004_automation.sql` with a `cleanup_expired_data()` Postgres function (SECURITY DEFINER, language plpgsql):
   - DELETE FROM price_snapshots ps USING monitored_cards mc WHERE ps.monitored_card_id = mc.id AND ps.recorded_at < now() - (mc.retention_days || ' days')::interval
   - GET DIAGNOSTICS snap_count = ROW_COUNT
   - DELETE FROM notifications n USING monitored_cards mc WHERE n.monitored_card_id = mc.id AND n.sent_at < now() - (mc.retention_days || ' days')::interval
   - GET DIAGNOSTICS notif_count = ROW_COUNT
   - RETURN QUERY SELECT snap_count, notif_count (both bigint)
   - Return type: TABLE(deleted_snapshots bigint, deleted_notifications bigint)

   - Log summary: "Deleted X snapshots and Y notifications older than retention" (per user decision)
   - Log elapsed time
   - main().catch() with Fatal error handler and process.exit(1)
